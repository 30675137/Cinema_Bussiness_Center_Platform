## 1. 功能定义

### 1.1 核心功能

库存台账是库存管理系统的核心模块，提供实时库存状态的查询、展示和管理能力，支持以下核心能力：

| 功能 | 描述 | 示例 |
|------|------|------|
| **多维库存查看** | 按仓库/门店和SKU维度展示库存状态 | 门店A可乐库存：50瓶 |
| **库存指标展示** | 显示现存、预占、可用、在途、安全库存等指标 | 现存100，预占30，可用70 |
| **多条件筛选** | 支持SKU关键词、门店、类目、品牌、库存状态组合筛选 | 筛选"低于安全库存"的酒水类商品 |
| **库存状态预警** | 自动计算并标识库存状态（缺货/低库存/正常/超库存） | 红色标签显示"缺货" |
| **库存调整操作** | 支持盘盈、盘亏、报损三种调整类型 | 盘点发现多出5瓶，录入盘盈 |
| **数据导出** | 支持将筛选结果导出为Excel格式 | 导出本月库存报表 |

### 1.3 典型应用场景

#### 场景一：日常库存巡检

| 角色 | 操作 | 系统功能 |
|------|------|----------|
| 店长 | 查看门店库存状态 | 筛选本门店所有SKU |
| 店长 | 关注低库存商品 | 状态筛选"低于安全库存" |
| 店长 | 发起补货申请 | 查看在途库存，判断是否需要追加 |

#### 场景二：盘点差异处理

| 步骤 | 操作 | 系统功能 |
|------|------|----------|
| 1 | 实物盘点发现差异 | 对比台账数量与实盘数量 |
| 2 | 录入库存调整 | 选择盘盈/盘亏类型，输入调整数量 |
| 3 | 填写调整原因 | 选择原因字典：过期/溢洒/操作错误等 |
| 4 | 提交审批 | 调整单进入审批流程 |

#### 场景三：销售出品验证

| 场景 | 查询目的 | 关注指标 |
|------|----------|----------|
| 下单前 | 验证库存充足 | 可用库存 ≥ 订单需求量 |
| 出品时 | 确认预占成功 | 预占数量已锁定 |
| 出品后 | 验证扣减正确 | 现存库存已减少 |

## 3. 核心数据模型

### 3.1 库存台账实体

```
┌─────────────────────────────────────────────────────────────────────────┐
│                       CurrentInventory (库存台账)                        │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  基础信息                                                               │
│  ├── id: string              # 台账记录ID                               │
│  ├── skuId: string           # SKU ID                                   │
│  ├── storeId: string         # 门店/仓库ID                              │
│  └── lastUpdated: string     # 最后更新时间                             │
│                                                                         │
│  库存指标                                                               │
│  ├── onHandQty: number       # 现存数量（实际库存）                     │
│  ├── availableQty: number    # 可用数量（现存 - 预占）                  │
│  ├── reservedQty: number     # 预占数量（已被订单锁定）                 │
│  ├── inTransitQty: number    # 在途数量（已采购/调拨未到货）            │
│  ├── damagedQty: number      # 损坏数量                                 │
│  └── expiredQty: number      # 过期数量                                 │
│                                                                         │
│  阈值设置                                                               │
│  ├── safetyStock: number     # 安全库存                                 │
│  ├── minStock: number        # 最小库存                                 │
│  ├── maxStock: number        # 最大库存                                 │
│  └── reorderPoint: number    # 补货触发点                               │
│                                                                         │
│  成本信息                                                               │
│  ├── averageCost: number     # 平均成本                                 │
│  └── totalValue: number      # 库存总价值                               │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.2 库存状态计算

| 状态 | 计算规则 | 颜色标识 |
|------|----------|----------|
| **缺货** | 可用数量 ≤ 0 | 🔴 红色 |
| **低于安全库存** | 0 < 可用数量 < 安全库存 | 🟠 橙色 |
| **正常** | 安全库存 ≤ 可用数量 ≤ 最大库存 | 🟢 绿色 |
| **超库存** | 可用数量 > 最大库存 | 🔵 蓝色 |

### 3.3 库存指标公式

```

│                                                                         │

│                                                                         │
```

## 6. 最佳实践方法

### 6.1 库存管理原则

#### 原则一：数据准确性优先

```
推荐做法：
  ├── 定期盘点，及时发现差异
  ├── 差异必须填写原因，留痕可追溯
  └── 审批后才能调整库存

不推荐做法：
  ├── 长期不盘点，差异累积
  ├── 随意调整库存，无审批流程
  └── 调整原因填写"其他"，无法分析
```

#### 原则二：合理设置安全库存

| SKU类型 | 安全库存建议 | 原因 |
|---------|--------------|------|
| 高频热销品 | 日均销量 × 3-5天 | 避免缺货影响销售 |
| 普通商品 | 日均销量 × 7天 | 保证补货周期 |
| 长尾商品 | 日均销量 × 14天 | 采购频次低 |
| 季节性商品 | 动态调整 | 旺季提高，淡季降低 |

#### 原则三：及时处理异常库存

| 异常类型 | 处理方式 | 时效要求 |
|----------|----------|----------|
| **缺货** | 紧急采购/调拨 | 24小时内 |
| **低库存** | 发起补货申请 | 48小时内 |
| **超库存** | 分析原因，考虑促销/调拨 | 1周内 |
| **过期库存** | 报损出库 | 发现即处理 |

### 6.2 日常操作流程

```
每日库存巡检流程
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

早班（9:00）
├── 1. 查看库存台账，筛选"缺货"状态商品
├── 2. 确认缺货商品的在途库存情况
├── 3. 对无在途的缺货品发起紧急补货
└── 4. 更新库存预警通知群

晚班（21:00）
├── 1. 汇总当日库存变动情况
├── 2. 检查预占库存是否正常释放
├── 3. 处理当日盘点差异单
└── 4. 确认明日到货计划

每周（周一）
├── 1. 导出上周库存报表
├── 2. 分析盘亏/报损原因分布
├── 3. 调整安全库存阈值
└── 4. 优化补货策略
```

## 7. 实际应用案例

### 7.1 案例：酒吧日常库存管理

#### 场景描述

某影院吧台每日需要管理约50种酒水饮料的库存，包括威士忌、啤酒、饮料等。

#### 操作流程

**第一步：每日开店检查**

```
操作路径：库存管理 > 库存台账

筛选条件：
├── 门店：影院吧台
├── 类目：酒水
└── 库存状态：缺货 + 低于安全库存

检查结果：
┌─────────────────────────────────────────────────────────────┐
│ SKU编码   │ 名称         │ 现存  │ 可用  │ 安全库存 │ 状态   │
├─────────────────────────────────────────────────────────────┤
│ JK001     │ 杰克丹尼     │ 350ml │ 350ml │ 700ml   │ 低库存 │
│ HL001     │ 喜力啤酒     │ 0瓶   │ 0瓶   │ 24瓶    │ 缺货   │
│ KL001     │ 可乐         │ 500ml │ 350ml │ 1000ml  │ 低库存 │
└─────────────────────────────────────────────────────────────┘
```

**第二步：处理缺货商品**

```
对于喜力啤酒（缺货）：
├── 1. 点击"详情"查看在途库存
├── 2. 发现有48瓶在途（主仓调拨中）
├── 3. 预计下午到货，暂不处理
└── 4. 在POS系统标记"暂时缺货"

对于杰克丹尼（低库存）：
├── 1. 当前350ml，安全库存700ml
├── 2. 日均消耗约200ml
├── 3. 预估可支撑1.5天
├── 4. 发起调拨申请：从主仓调2瓶（1400ml）
```

**第三步：营业结束盘点**

```
威士忌盘点：
├── 账面库存：1050ml
├── 实际盘点：980ml
├── 差异：-70ml（约1.5杯用量）
├── 原因分析：可能是制作损耗未记录
└── 处理：录入盘亏，原因选择"制作损耗"

操作路径：
库存台账 > 杰克丹尼 > 库存调整
├── 调整类型：盘亏
├── 调整数量：70ml
├── 调整原因：制作损耗
└── 备注：营业结束盘点发现
```

### A. 数据库表结构

```sql
-- 库存台账表
CREATE TABLE current_inventory (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  sku_id UUID NOT NULL REFERENCES skus(id),
  store_id UUID NOT NULL REFERENCES stores(id),
  on_hand_qty DECIMAL(10,3) NOT NULL DEFAULT 0,
  reserved_qty DECIMAL(10,3) NOT NULL DEFAULT 0,
  in_transit_qty DECIMAL(10,3) NOT NULL DEFAULT 0,
  damaged_qty DECIMAL(10,3) NOT NULL DEFAULT 0,
  expired_qty DECIMAL(10,3) NOT NULL DEFAULT 0,
  safety_stock DECIMAL(10,3) NOT NULL DEFAULT 0,
  min_stock DECIMAL(10,3) NOT NULL DEFAULT 0,
  max_stock DECIMAL(10,3) NOT NULL DEFAULT 0,
  reorder_point DECIMAL(10,3) NOT NULL DEFAULT 0,
  average_cost DECIMAL(10,4),
  total_value DECIMAL(14,2),
  last_updated TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  CONSTRAINT uk_inventory_sku_store UNIQUE (sku_id, store_id)
);

-- 库存交易流水表
CREATE TABLE inventory_transactions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  sku_id UUID NOT NULL REFERENCES skus(id),
  store_id UUID NOT NULL REFERENCES stores(id),
  transaction_type VARCHAR(30) NOT NULL,
  quantity DECIMAL(10,3) NOT NULL,
  unit_cost DECIMAL(10,4),
  total_cost DECIMAL(14,2),
  stock_before DECIMAL(10,3) NOT NULL,
  stock_after DECIMAL(10,3) NOT NULL,
  source_type VARCHAR(30) NOT NULL,
  source_id UUID,
  source_document VARCHAR(50),
  operator_id UUID NOT NULL,
  transaction_time TIMESTAMP WITH TIME ZONE NOT NULL,
  remarks TEXT,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```
